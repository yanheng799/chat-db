# 语义匹配 + 单步查询核心链路（Phase 5）— 规格细化

- **slug**：`2026-06-19-semantic-matching-pipeline`
- **状态**：refining 完成（第 5 轮，核心行为已定，待 `team-spec-review` 复查）
- **基线**：设计文档 §八 语义匹配系统 + §十 安全与审核机制 + §三 3.1 正常流程；开发计划 Phase 5（5.1–5.5）
- **依赖**：Phase 2（语义描述 + 推断外键）、Phase 3（知识库：向量检索 + 图谱查询 + 知识库刷新）、Phase 4（值标准化器 + 值映射中心）
- **关联决策**：3.4 热词词典由本 Phase 构建（作为语义匹配器第 1 层）

## 需求复述

把 Phase 1-4 的成果串联成**首个可演示的端到端查询链路**：用户自然语言输入 → 语义匹配（四层递进识别字段）→ 值标准化（把口语值转成 SQL 可用值）→ 生成只读 SQL → 安全校验 → 执行 → 自然语言结果总结 → 返回用户。这是「首次可演示里程碑」。

## 问题与价值

- **痛点**：Phase 1-4 各自建了能力（元数据提取、语义描述、知识库、值标准化），但还没有一条链路把它们串成实际可用的查询。目前系统无法接收自然语言、无法返回查询结果。
- **价值**：第一个端到端 NL→SQL→结果的可演示闭环——能输入「昨天的订单总数」并得到真实答案。这是整个系统从「基建」到「可用」的转折点。
- **触发**：用户通过 Chat UI 发起自然语言查询。

## 用户与场景

1. 作为**查询用户**，我说「查一下昨天的订单总数」，系统自动理解意图、匹配字段、标准化时间、生成 SQL、执行、返回「昨天共有 128 笔订单」。
2. 作为**查询用户**，我说「上个月已完成的大额订单有哪些」，系统匹配字段、标准化时间和枚举值、检测「大额」→ 追问确认金额阈值。
3. 作为**查询用户**，当语义匹配置信度低或 LLM 兜底触发时，系统请求人工审核确认后再执行。

## 术语

| 术语 | 定义 |
|------|------|
| 语义匹配 | 把用户口语/行业黑话映射到具体的表.列（四层递进：热词词典→行业词库→向量检索→LLM 兜底）。 |
| 热词词典（3.4） | Phase 3 延期、Phase 5 构建的「热词→字段」确定性映射字典（含锁定业务指标公式，如「销售额 = SUM(price*quantity)」）。 |
| 单步查询 | 无需跨表 JOIN 或多步路由的单 SQL 查询。多步查询延到 Phase 6（LangGraph 编排）。 |

## 范围内（本规格）

- **3.4 热词词典（收编）**：打包内置热词→字段映射字典 + 业务指标公式模板，V1 无 CRUD UI（代码/配置维护）。
- **5.1 四层语义匹配**：热词词典 → 行业词库 → 向量检索（Phase 3 field_descriptions）→ LLM 兜底（标注 source=llm_fallback，触发强制审核）。
- **5.2 单步 SQL 生成**：LLM prompt 注入 schema 上下文 + 标准化值 + 语义匹配结果，约束只读/LIMIT/禁止 SELECT *。
- **5.3 安全校验**：黑名单（禁止 DML/DDL）+ 白名单（LIMIT ≤ 1000、禁止 SELECT *）+ 语法校验。
- **5.4 SQL 执行**：只读连接 + 超时控制 + 结果捕获。
- **5.5 单步查询完整串联**：编排上述步骤成一条链路，含审核阻断点（若需要）。

## 范围外 / 延期

- 多步查询路由 / 多 Agent 编排（Phase 6 LangGraph）。
- 错误自愈（元数据过期自动重试）—— Phase 7。
- 管理端审核策略配置 UI（Phase 10）。
- 热词词典 / 行业词库的 CRUD UI（Phase 10）。
- Chat UI 前端（Phase 10）。
- 查询历史 / 结果缓存 / 用户画像（Phase 11/用户特征系统）。

## 行为规格 — 管道顺序（已定）

```
用户输入
  → 时间标准化（前置提取；Phase 4 parse_time）
  → 语义匹配（四层递进：热词词典→行业词库→向量检索→LLM 兜底）
  → 枚举/区域/名称标准化（对每个已识别字段调 Phase 4 标准化器；依赖字段上下文）
  → SQL 生成（LLM prompt 注入 schema + 标准化值 + 匹配结果）
  → 安全校验（黑名单 + 白名单 + 语法）
  → [审核阻断 — 若 LLM 兜底被触发 或 量词 need_confirm]
  → SQL 执行（只读 + 超时）
  → 自然语言结果总结（LLM）
  → 返回用户
```

## 行为规格 — 审核阻断（已定）

- V1 审核 = **用户确认**（复用 Phase 4 `need_confirm` 模式，无管理端审核队列）。
- 语义匹配 LLM 兜底结果：`matched_by="llm_fallback"`、`need_confirm=True`。
- Phase 5 串联中的「审核阻断」步骤：**收集本查询中所有 `need_confirm=True` 的项**（含语义匹配的 LLM 推断结果 + Phase 4 标准化器的未解值 + 量词），**一次性批量展示给用户确认**。
- 用户确认 → SQL 执行；用户拒绝 → 返回候选/让用户修正。
- 管理端审核策略配置 UI / 审核仪表盘 → 延 Phase 10。

## 行为规格 — 热词词典 V1 规模（已定）

- **热词（第 1 层）**：10–15 条精选常见电商/ERP 查询术语 → 字段/聚合映射（如「销售额→SUM(price*quantity)」「订单数→COUNT(*)」「客单价→SUM(amount)/COUNT(DISTINCT customer_id)」）。
- **公式模板**：2–3 条锁定业务指标（如 `毛利率=(SUM(price)-SUM(cost))/SUM(price)`），`locked=true`，禁止 LLM 动态编造。
- **行业词库（第 2 层）**：5–10 条通用行业术语（如「GMV→销售额」「SKU→库存单位」），用于用户在口语中混用行业黑话时做翻译。
- 大部分匹配走第 3 层（向量检索）和第 4 层（LLM 兜底）；热词+行业词是加速器，V1 不求全。

## 行为规格 — 安全/SQL 故障处理（已定）

- **安全校验失败**：透明告知用户原因（如「SQL 含 DROP 关键词」），同时将失败信息作为上下文**回传 LLM 重生成 SQL**（最多 1 次）。重试后仍失败 → 向用户报告、不执行。
- **SQL 执行超时/错误**：V1 **只捕获错误并告知用户**（含错误原因和原始 SQL），**不做自愈**（元数据过期重试 / 自动修正 SQL / 加 LIMIT 再试延到 Phase 7）。
- **主动防御**：黑名单（`INSERT|UPDATE|DELETE|DROP|TRUNCATE|ALTER|EXEC|INTO OUTFILE|LOAD|SLEEP|BENCHMARK|@@version`）、白名单（`LIMIT ≤ 1000`，禁止 `SELECT *`）、语法校验（`sqlparse`）。

## 验收口径

**应通过**：
- 「查一下昨天的订单总数」→ 端到端返回正确计数（时间前置→语义匹配→标准化→SQL 生成→安全校验→执行→结果总结）。
- 语义匹配命中热词「销售额」→ `matched_by="hot_word"`（第 1 层确定性）。
- 语义匹配经向量检索命中 `orders.status`（第 3 层）。
- LLM 兜底被触发 → `matched_by="llm_fallback"`、`need_confirm=True`，与 Phase 4 的未解值合并批量展示给用户确认。
- SQL 安全校验命中黑名单 → 告知用户理由 + LLM 重生成一次 → 若仍失败 → 不执行。
- SQL 执行超时 → 捕获错误、告知用户，不重试。
- 用户体验：一次查询请求返回结果（或追问确认）≤ 30 秒（含 LLM 调用与向量检索）。

**应失败 / 边界**：
- 空查询「」→ 拒绝，告知用户输入查询内容。
- SQL 生成结果不含 SELECT → 安全校验拦截。
- 目标数据源未激活或不可达 → 告知用户当前无可用数据源。
- 语义匹配全四层都未匹配到任何字段 → 告知用户无法理解查询、请换种问法。

## 轻量风险扫尾

- **无 P0**。
- **P1**：LLM 调用次数（Phase 5 单次查询可能调用 3 次 LLM——语义匹配第 4 层兜底、SQL 生成、安全失败重生成、结果总结——调用量比 Phase 2 L2 大得多，成本与延迟需要管控）。建议：在实现期对单次查询的 LLM 调用次数设上限（如 5 次），超限降级。
- **P2**：向量检索延迟（Phase 3 field_descriptions 是增量 upsert，查询时需 `consistency_level="Strong"` → 可能增加延迟）。V1 先默认 Strong；性能优化延 Phase 11。
- **P2**：热词词典初始内容的具体列表需在实现期确定（PRD 给一个初始清单）。
- **P2**：SQL 生成的「单步」限制——如何确保 LLM 只生成单步查询而非子查询/JOIN？V1 依赖 prompt 约束 + 安全校验拦截多表 JOIN（可能过于严格），需要在实现期平衡。
- **P3**：结果总结的 LLM 是否外发原始数据行——Phase 2 L2 治理禁止外发原始行。但 Phase 5 结果总结需要把查询结果发给 LLM 做自然语言包装。这违反了 Phase 2 的「不下发原始行」原则——需要显式决策：结果总结的 LLM 调用是否受同一治理约束？V1 建议：结果总结**只发聚合值**（计数/汇总），不发行级数据；若查询返回行级结果且用户要求总结，系统告知「不建议发送原始数据给 LLM」，改为直接返回表格。
