## 需求复述

Phase 5 让单步查询跑通了，但真实业务问题往往需要拆解——「对比本月和上月的销售额，按地区排名」需两个时间窗口 + 跨表 JOIN + 聚合。Phase 6 用 LangGraph 状态图判断单步/多步路由，多步时查图谱做 JOIN 路径规划、生成子任务 DAG、按顺序执行子查询并传递中间结果。复用 Phase 5 的单步模块作为多步 DAG 中的子执行单元。

## 问题与价值

- **痛点**：Phase 5 只能回答单表单步问题，含 JOIN 或需要拆解的用户问题（对比/拆步骤/多表）直接报错。用户的自然语言天然是多步的，系统必须能拆。
- **价值**：让系统从「单步查询玩具」升级为「能拆解复杂问题的 Agent」——LangGraph 状态图让流程可追踪、可中断、可恢复。

## 用户与场景

1. 作为**查询用户**，我说「对比本月和上月的销售额」，系统自动判定多步、生成两个子查询计划、分别执行、合并结果并做对比总结。
2. 作为**查询用户**，我说「包含客户名的已完成订单」，系统经图谱短路径发现 orders↔customers 需 JOIN，生成多步计划。
3. 作为**查询用户**，若多步执行中某子查询失败，系统记录失败步骤、跳过继续执行其余子任务，最终汇总含失败标记的结果。

## 术语

| 术语 | 定义 |
|------|------|
| 状态图（State Graph） | LangGraph 定义的多 Agent 状态机，含节点（Agent）与条件边（路由规则）。 |
| DAG 执行器 | 对计划生成产生的子任务有向无环图做拓扑排序后按序执行、子结果传递。 |
| 单步/多步路由 | 根据用户查询是否包含多步信号（对比/多表/拆步骤）决定走 Phase 5 单步管道还是 Phase 6 多步计划。 |

## 范围内（本规格）

- **6.1 LangGraph 状态图**：State Schema + 节点定义 + 条件边（单步/多步路由、审核阻断、错误处理）。
- **6.3 计划生成 Agent**：规则检测多步信号 → 查询图谱 JOIN 路径（Phase 3 `shortest_join_path`）→ 拆解子任务 DAG。
- **6.6 多步查询串联**：DAG 执行器（拓扑排序 → 执行子查询 → 子结果传递）→ 最终结果合并 + LLM 摘要。
- **复用 Phase 5**：联合分析 = 编排 Phase 5 `run_single_step` 作为子执行单元；执行+审核 = 复用 Phase 5 安全校验+SQL 执行器。

## 行为规格 — 路由判定（已定）

- **在状态图内**：首节点 `classify_query` 做两层纯规则判定（不调 LLM）。
  1. **规则检测**：用 `MULTI_STEP_PATTERNS` 正则扫用户原文（对比/比较/变化/先…再/然后/同时查）。命中 → 路由到计划生成（多步）。
  2. **图谱检测**：若语义匹配识别到 >1 表，且 Phase 3 `shortest_join_path` 确认存在 JOIN 路径 → 路由到计划生成。
  3. 以上均不满足 → 路由到 Phase 5 `run_single_step`（单步快速通道）。

## 行为规格 — DAG 故障恢复（已定）

- **best-effort 继续**：子任务失败 → 标记该节点 failed，跳过其后继依赖节点（无结果可传），与其不相关的分支继续执行。
- **部分结果呈现**：成功子结果合并展示，失败子任务在结果中标注原因（如「本月销售额查询失败：数据库超时」）。
- **不重试**（V1 不自愈——Phase 7 统一做错误自愈）。

## 行为规格 — 图谱集成与摘要治理（已定）

- **图谱集成**：直接调 Phase 3 `shortest_join_path(graph_store, ds_id, table_a, table_b)`，返回 `[{from_table, from_column, to_table, to_column, type, confidence}]`。不传 `min_confidence`（接受所有推断边，计划侧再判断）。
- **LLM 摘要治理**：与 Phase 5 一致——**仅对聚合子结果（COUNT/SUM/AVG）做 LLM 自然语言包装**；子查询的行级结果不传入 LLM（仅展示行数/统计信息）。

## 验收口径

**应通过**：
- 「查一下昨天的订单总数」→ 路由判定为单步 → 由 Phase 5 `run_single_step` 执行并返回。
- 「对比本月和上月的销售额」→ 路由判定命中多步信号 → 进入计划生成 → 拆为两个子查询 DAG → 顺序执行 → 合并结果 + LLM 对比摘要。
- 「包含客户名的已完成订单」→ 语义匹配识别 orders+customers → 图谱确认 JOIN 路径 → 计划生成含 JOIN 子查询。
- DAG 中某子查询失败 → 不阻断其他不相关子任务 → 成功结果正常展示 + 失败任务标注原因。
- LLM 摘要仅对聚合结果做自然语言包装，行级结果不传入摘要 LLM。

**应失败 / 边界**：
- 空输入 → 拒绝。
- 路由判定均不触发、Phase 5 也无法执行 → 返回错误。
- DAG 全部子任务失败 → 返回错误（无成功结果可展示）。
- 图谱中找不到 JOIN 路径的两表 → 计划生成只出单表子任务（不强制多步）。

## 轻量风险扫尾

- **无 P0**。
- **P1**：LangGraph 编排引入额外延迟与 LLM 调用（相比 Phase 5 单步多 1-2 次 LLM 调用：计划生成 + 摘要）。需管控超时与调用上限（沿用 Phase 5 `pipeline_llm_max_calls` 模式）。
- **P2**：路由判定的 MULTI_STEP_PATTERNS 可能漏检或误判（正则覆盖不够全）。V1 用设计文档的预设 pattern + 图谱检测兜底；Phase 6/7 联调时迭代 pattern。
- **P2**：DAG 依赖解析可能遇到循环依赖（计划生成出错时）。V1 对生成的 DAG 做环检测，发现环 → 拒绝执行 + 报错。
- **P2**：多步查询的中间结果传递（子任务 1 的结果传给子任务 2 作为参数）——数据边界需在实现期明确（传行数/聚合值/还是全量行？V1 仅传聚合统计值，不传原始行）。
