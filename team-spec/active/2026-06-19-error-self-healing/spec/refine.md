# 错误自愈系统（Phase 7）— 规格细化

- **slug**：`2026-06-19-error-self-healing`
- **状态**：refining 完成（第 4 轮，核心行为已定，待 `team-spec-review` 复查）
- **基线**：设计文档 §五 错误自愈系统；开发计划 Phase 7（7.1–7.5）
- **依赖**：Phase 1 元数据同步引擎（已实现）、Phase 3 向量检索+图谱（PR #22）、Phase 5 单步管道（PR #36）、Phase 6 多步编排（PR #40）

## 需求复述

SQL 执行失败不是终点——作为有状态的 Agent 系统，Phase 7 在 SQL 执行失败后分类错误类型，对元数据过期自动触发同步并重试，对 SQL 语法/类型错误做 LLM 多策略重写修复，对跨表缺字段查向量库+图谱做自动 JOIN。每次修复后重试查询，最多 N 次。成功修复的案例记录到错误学习闭环，反馈回元数据和匹配模型。

## 问题与价值

- **痛点**：Phase 5/6 遇到 SQL 执行错误直接报错返回用户，不做自动修复。但设计文档核心认知——「表名/字段名错误 99% 来自元数据过期，重点在同步机制」+「每次错误都是学习机会」。
- **价值**：从「报错中断」升级为「自动修复→重试→成功」的弹性系统。减少用户因元数据过期/SQL 小错误而被阻塞的概率。

## 范围内（本规格）

- **7.1 错误分类器**：解析 DB 错误消息（postgres/MySQL），归类为 table_not_found / column_not_found / sql_syntax_error / type_mismatch / other。
- **7.2 元数据过期处理**：table_not_found / column_not_found → 触发 Phase 1 同步引擎（手动同步该表/全量）→ 重试查询。
- **7.3 跨表字段自愈**：column_not_found → 向量库全局搜索 → 过滤关联表 → 查图谱 JOIN 路径 → 自动 JOIN 或返回候选。
- **7.4 SQL 重写修复**：sql_syntax_error / type_mismatch → LLM 多策略修复（附加错误信息）→ 安全校验 → 重试（最多 3 次，超限降级为告知用户）。
- **7.5 错误学习闭环**：成功修复的模式记录——元数据同步修复 → 更新向量库+图谱；自动 JOIN → 不记录（JOINS_WITH 频次依赖查询历史，V1 暂不累积）；SQL 重写成功 → 记录错误模式供后续参考。

## 范围外 / 延期

- 跨库字段映射（不同数据源之间的字段同义——Phase 10）。
- `JOINS_WITH` 频次实时累加（需要查询历史统计基础设施——Phase 11）。

## 行为规格 — 触发与重试（已定）

- **独立 Agent**：Phase 5/6 管道执行 SQL 失败后，不直接报错返回——将错误信息（原始 SQL + 错误消息 + 查询上下文）写入错误事件，**自愈 Agent 异步接管**修复→重试流程。Phase 5/6 管道等待自愈结果（超时上限 60s）后返回最终结果。
- **最多 3 次**：修复→重试循环最多 3 次，每种修复策略最多尝试一次（元数据同步、LLM 重写、跨表 JOIN，去重不重复）。
- **transparent**：不逐次打断用户——最终告知「已自动修复并查询」（成功）或「已尝试修复仍失败」（全部失败）。
- **LLM 调用计入 `pipeline_llm_max_calls`** 共享池。

## 行为规格 — 跨表自愈集成（已定）

- **共用 LangGraph 状态图**：自愈 Agent 是 Phase 6 状态图的一个条件分支（`healing_agent` 子图）：SQL 执行失败→条件边路由到 healing 子图→修复后回到执行节点重试。
- 跨表字段自愈复用 Phase 6 `execute_dag`（拓扑排序+执行+失败断依赖+结果合并）。
- 不再另建独立 Agent 进程或 mini-DAG 解析器。

## 行为规格 — 学习闭环存储（已定）

- **PG 新表 `healing_records`**：`error_type, original_sql, fix_type, fix_sql, success, timestamp`。仅追加记录。V1 不做相似错误检索/匹配。
- 对 metadata_sync/auto_join 修复的反馈直接调 Phase 3 向量/图谱更新，不依赖存储。

## 验收口径

**应通过**：
- SQL 执行报 `table not found` → 分类器判断 `table_not_found` → 触发元数据同步 → 同步完成后重试 SQL → 重试成功返回结果。
- SQL 语法错误 → `sql_syntax_error` → LLM 重写 SQL → 安全校验 → 重试成功（或 3 次全失败后告知用户）。
- 字段不存在且跨表候选命中 → 向量搜索+图谱 JOIN → 自动构建 JOIN 查询 → 重试成功。
- 自愈最多重试 3 次；全部失败 → 告知用户，记录 `healing_records` 失败日志。

**应失败/边界**：
- 错误类型为 `other`（无法分类）→ 直接告知用户，不做自愈。
- 元数据同步本身失败 → 不重试，告知用户「元数据同步失败，请手动检查」。

## 轻量风险扫尾

- **无 P0**。
- **P1**：自愈 Agent 的 60s 等待超时（Phase 5/6 等待自愈结果）——如果同步耗时超过 60s 用户可能在管道中一直被挂起。V1 60s 是硬超时，超时后返回部分修复结果或用户可取消。
- **P2**：错误分类器只基于错误消息字符串解析（`pgcode` / `errno`），不同数据库版本的错误消息可能不匹配——V1 按 postgres/MySQL 写正则 + 兜底 `other`。
- **P2**：跨表字段自愈依赖 Phase 3 向量库+图谱均正常——如果 Phase 3 不可用（如向量检索超时），自愈会额外失败。V1 接受，Phase 3 基础设施已就绪。
- **P2**：`healing_records` 只追加不修剪，长期可能膨胀——V1 接受，Phase 11 加清理策略。
