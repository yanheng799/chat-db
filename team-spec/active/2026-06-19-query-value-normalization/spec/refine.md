# 查询值标准化（Phase 4）— 规格细化

- **slug**：`2026-06-19-query-value-normalization`
- **状态**：refining 完成（第 5 轮，核心行为已定，待 `team-spec-review` 复查）
- **基线**：设计文档 §四 查询值标准化系统；开发计划 Phase 4（4.1–4.6）
- **依赖**：Phase 1 元数据（`detected_enum_values` 可作枚举种子）；Phase 2 语义描述（可辅助名称匹配）；Phase 3 知识库（向量检索可用于名称匹配 strategy 7）
- **关联决策**：Phase 3 延期项 3.5 值映射中心已收编入本规格

## 需求复述

用户的自然语言查询会带口语化的**值表述**——「上周」「大客户」「华东」「已完成」——这些 SQL 不能直接消费。Phase 4 把查询里的时间、枚举值、区域/地名、名称简称、数值范围**标准化为数据库可直接使用的值**，返回结构化结果（含置信度、是否需要用户确认）。

## 问题与价值

- **痛点**：用户用口语说「上个月北京已完成的订单」，但 SQL 要求 `WHERE date BETWEEN ... AND city IN (...) AND status = 'completed'`。没有标准化，Phase 5/6 无法生成正确 SQL。
- **价值**：把口语值变成结构化值，让 SQL 生成 Agent 可直接消费；对低置信度场景要求用户确认、降低错误风险。

## 用户与场景

1. 作为**查询用户**，我说「上个月上海大额的已支付订单」，系统能自动把「上个月」→ 日期范围、「上海」→ city IN ('上海')、「已支付」→ status='paid'，「大额」→ 返回追问（模糊量词不自动量化）。
2. 作为**管理员**，我通过 API 管理枚举别名、区域字典、名称简称的映射关系，以便系统正确标准化。

## 术语

| 术语 | 定义 |
|------|------|
| 值标准化 | 把用户输入的值表述（口语、简称、模糊量词）转为数据库可用的结构化值。 |
| 值映射中心（3.5） | Phase 3 延期、Phase 4 收编的枚举别名映射 / 区域字典 / 名称简称映射的 CRUD 存储。 |
| 标准化流水线 | Phase 5 先做语义匹配识别字段，然后 Phase 4 针对已识别的字段做值标准化，最终输出 `NormalizedValue`。 |

## 范围内（本规格）

- **4.1 统一数据结构**：`NormalizedValue`（original / normalized / value_type / db_representation / confidence / matched_by / need_confirm / alternatives）。
- **4.2 时间表述标准化**：相对时间、绝对时间、（固定日期型）自定义周期。
- **4.3 枚举值映射**：5 策略递进（字典表→精确→别名→编辑距离≥0.7→LLM 兜底 confidence>0.85）。
- **4.4 区域/地名处理**：粒度自适应（区域/城市/区级）+ 层级包含统一为 SQL IN。
- **4.5 名称简称匹配**：7 策略递进（精确→简称→别名→拼音→关键字→编辑距离≥0.7→向量语义）。
- **4.6 数值范围处理**：模糊量词检测 → 追问，不自动量化。
- **收编的 3.5 值映射中心**：枚举别名映射 / 区域字典 / 名称简称映射的仓库存储 + CRUD API + 初始化/种子数据策略。

## 范围外 / 延期

- **3.4 热词词典**（含锁定业务指标公式 / 固定日期型周期配置）—— 继续独立，归 Phase 3.b 或 Phase 5 前置。
- 语义匹配 / SQL 生成 / 多步编排（Phase 5/6）。
- 管理端 UI（Phase 10）；本规格只做后端 API。
- 模糊量词的自定义阈值定义（如「大额 > 10000」这类规则不由 Phase 4 定义，追问让用户明确）。
- LLM 兜底的外发治理（standardization LLM 调用属 Phase 5 链路的一部分，治理约束在 Phase 5 规格定）。

## 行为规格 — 标准化流水线位置与接口（已定）

- **后置为主**：Phase 5 语义匹配先识别字段（表/列 + 口语值表述），然后将 `(field_context, raw_value)` 交给 Phase 4 标准化。枚举/区域/名称标准化**依赖字段上下文**（如 `orders.status` 的枚举别名），所以必须在字段识别之后。
- **时间可前置**：时间标准化不依赖字段上下文（「上个月」可直接解析为日期范围），允许在语义匹配之前独立提取，用于限定查询时间范围。
- **Phase 6 消费**：Phase 4 返回 `NormalizedValue` 列表（含 `db_representation`、`confidence`、`need_confirm`），Phase 6 SQL 生成直接使用。
- **无字段上下文的值**：若 Phase 5 未识别到字段、或字段无映射数据，标准化失败不抛错——返回 `NormalizedValue` 标记 `need_confirm=True`，Phase 6 遇到未标准化值时可回退追问。

## 行为规格 — 值映射中心数据隔离（已定）

- **隔离粒度**：枚举别名表、名称简称表、区域字典表**均按 `data_source_id` 命名空间隔离**（不同业务库独立管理、无跨源冲突）。
- **区域种子**：系统提供一份**预置的中国行政区划种子数据**（省/市/区 + 层级关系），管理员激活新源时可一键导入到该源的区域字典中；导入后可按需增删自定义区域（如销售大区）。
- **CRUD API**：所有映射管理端点带 `data_source_id`，与元数据/知识库隔离模式一致。

## 行为规格 — 失败与追问策略（已定）

- **统一底层**：任何标准化器策略链用尽 → 返回 `NormalizedValue(need_confirm=True, db_representation=None, confidence=0)`。Phase 6 收集所有 `need_confirm` 值，**批量一次性**向用户展示追问（不逐字段打断），用户确认后重新生成 SQL。
- **枚举 LLM 兜底**：治理与 Phase 2 L2 一致——**仅用结构化信号**（字段名 + 已知枚举值列表 + 用户口语文本），**不下发原始业务数据行**。LLM 失败 → need_confirm。
- **名称全策略失败**：回退到目标库 `SELECT DISTINCT col FROM table WHERE col LIKE '%xxx%' LIMIT 10`（复用 `query_executor` + 只读 + 限流），结果作为 `alternatives` 供用户选择；LIKE 也失败 → need_confirm。
- **区域**：粒度自适应匹配失败 → need_confirm。
- **数值**：模糊量词检测 → 始终 need_confirm（不自动量化）。

## 行为规格 — 种子数据与维护（已定）

- **枚举别名**：首次学习完成后，从 Phase 1 `detected_enum_values` 自动采集为该数据源各列的枚举记录（value=原始值、display=原始值、aliases=空数组）；后续新增列也在学习后增量采集。管理员事后补充别名。
- **区域字典**：系统预置一份**中国行政区划 CSV（省/市/区 + 层级）**作为种子文件；激活新源时自动导入到该源的区域字典中。管理员可增删自定义区域。
- **名称简称**：无自动种子——管理员手工录入。
- **固定日期周期**：**代码常量**（`FIXED_DATE_PERIODS`），发小于预设列表；管理员可通过 CRUD API 增删。

## 验收口径

**应通过**：
- 用户输入「上个月上海已完成的订单」→ 时间标准化输出上月日期范围、枚举标准化输出 `orders.status='completed'`（经别名映射）、区域标准化输出 `city IN ('上海')`。
- 枚举别名匹配「完结」→ `completed`（别名表中存在映射）。
- 名称简称匹配「华为」→ `Huawei Technologies`（简称表中存在映射）。
- 区域「华东」→ 被识别为销售大区（管理员自定义区域），展开为下属省/市 SQL IN。
- 模糊量词「大额订单」→ `need_confirm=True`，Phase 6 追问「您定义的'大额'是多少？」。
- 枚举 LLM 兜底调用仅发结构化信号（字段名 + 枚举值列表 + 用户输入），**不含原始业务数据行**。
- 名称全策略失败 → 回退到目标库 `SELECT DISTINCT col ... LIKE '%xxx%' LIMIT 10`，结果作为 alternatives。
- 所有标准化器策略用尽 → 统一 `need_confirm=True`；Phase 6 批量一次性展示所有待确认项。

**应失败 / 边界**：
- 区域字典中不存在的城市名 → `need_confirm=True`。
- 枚举值在别名表和 LLM 兜底都匹配不到 → `need_confirm=True`。
- 名称 LIKE 回退返回 0 行 → `need_confirm=True`，alternatives 为空。
- 目标库不可达（名称 LIKE 失败）→ 静默跳过 LIKE，need_confirm（不阻断流水线）。
- 管理员删除某源数据源时，该源的映射数据同步清理（`CASCADE` 或手动）。

## 轻量风险扫尾

- **无 P0**。
- **P1**：枚举 LLM 兜底的外发数据治理（仍调用外部 LLM——与 Phase 2 L2 相同约束：仅结构化信号、不下发原始行）。Phase 2 治理决策已有先例，不阻塞 PRD。
- **P2**：名称 LIKE 回退需要连接目标业务库（Phase 4 首次引入目标库读路径）——权限/限流/超时沿用 Phase 2 `query_executor` 抽象，但增加了一类新的目标库查询场景，需在实现期显式标注。
- **P2**：区域种子 CSV 的时效性与准确性（行政变更）。V1 发货一份快照，后续由 Ops/管理员维护。
- **P2**：固定日期周期的跨年处理（如「双十一」跨年时范围解歧）。V1 仅固定日期、不含计算型周期（明确排除）。
- **P3**：名称简称的 7 策略递进顺序的准确性调优（拼音、编辑距离阈值）需实际语料打磨，V1 先用默认阈值。
