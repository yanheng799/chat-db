# 自查与评审清单

## 实现完成前自查

### 代码质量

- [ ] 无 `TODO`、`FIXME`、`pass`、`NotImplementedError`、`...` 占位符
- [ ] `ruff check src/` 无错误
- [ ] `ruff format --check src/` 无格式问题
- [ ] 无 `print()` 语句（使用 `structlog`）
- [ ] 无硬编码的密钥、URL 或路径

### 设计原则

- [ ] 新增行为通过扩展实现，未修改已有的分发器/策略（OCP）
- [ ] 外部依赖通过抽象接口访问（DIP）
- [ ] 模块保持单一职责（SRP）
- [ ] 异常使用自定义异常类，非内置异常

### 测试

- [ ] 对应 `test/test_{module}/` 下有测试文件
- [ ] 测试覆盖主要行为路径（正常、边界、异常）
- [ ] 外部服务已 mock（LLM、向量库、图数据库、Redis、PG）
- [ ] `pytest` 全部通过
- [ ] 测试先于实现编写（TDD Red-Green-Refactor）

### 配置

- [ ] 新增配置项已添加到 `.env.example`
- [ ] 配置通过 `pydantic-settings` 管理，无 YAML
- [ ] 新增包已添加到 `pyproject.toml` 的 `packages` 列表

### 构建

- [ ] `pip install -e .` 能正常安装（或 `uv sync`）
- [ ] 新增包可被正确 `import`

## 评审关注点

### 行为正确性

1. 是否满足 issue/需求描述中的验收条件
2. 边界情况是否处理（空输入、超长输入、非法类型）
3. 异步代码是否正确使用 `async/await`，无遗漏的 `await`

### 回归风险

1. 修改是否影响其他模块的公共 API
2. 配置变更是否向后兼容
3. 数据库相关变更是否考虑了迁移

### 安全性

1. 生成的 SQL 是否通过黑名单/白名单校验
2. 用户输入是否做了适当的净化
3. 错误信息是否泄漏了内部细节

### 可观测性

1. 关键操作是否有日志
2. 日志级别是否恰当（DEBUG/INFO/WARNING/ERROR）
3. 错误是否可追溯（有足够的上下文信息）

### 可维护性

1. 公共 API 是否简洁明确
2. 命名是否一致（与同模块其他代码风格统一）
3. 注释是否恰当（不多不少，解释"为什么"而非"是什么"）
4. 新增的包/模块是否与现有架构一致
