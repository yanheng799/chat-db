## 二、对话页 — 核心交互流程

### 2.1 状态机

对话页是系统的核心。用户发起一次查询的完整生命周期如下：

```
                         ┌──────────────────────────┐
                         │   IDLE (空闲)             │
                         │   显示空状态 / 历史消息    │
                         └──────┬───────────────────┘
                                │ 用户输入查询文本
                                ▼
                         ┌──────────────────────────┐
                         │   RUNNING (执行中)        │
                         │   禁用输入，显示进度条     │
                         └──────┬───────────────────┘
                                │ SSE 事件流
                    ┌───────────┼───────────┬──────────────┐
                    ▼           ▼           ▼              ▼
              need_confirm   result      error          done
                    │           │           │              │
                    ▼           ▼           ▼              ▼
         ┌──────────────┐ ┌────────┐ ┌────────┐  ┌──────────┐
         │ AWAIT_CONFIRM│ │SUCCESS │ │ FAILED │  │ COMPLETE │
         │ 显示确认卡片  │ │显示结果│ │显示错误│  │ 回到IDLE │
         └──────┬───────┘ └───┬────┘ └───┬────┘  └──────────┘
                │              │          │
      用户点击确认/取消        │          │
                │              │          │
                ▼              ▼          ▼
         回到 RUNNING     回到 IDLE   回到 IDLE
         (继续执行)                     (可重试)
```

### 2.2 SSE 事件处理规范

| SSE 事件类型 | 触发时机 | UI 响应 |
|-------------|---------|---------|
| `status` { message: "starting" } | 查询开始 | 显示系统气泡"思考中…"，进度条 Step1 active |
| `status` { message: "step_name" } | 阶段推进 | 更新进度条步骤状态，更新系统气泡文字 |
| `need_confirm` { items: [...] } | 审核门触发 | 替换系统气泡为确认卡片，Step1/2 done |
| `result` { data: {columns, rows} } | SQL 执行成功 | 替换系统气泡为结果表格，所有步骤 done |
| `error` { detail: "..." } | 执行失败 | 显示红色错误信息，所有步骤 done |
| `done` | 查询完全结束 | 所有步骤 done，恢复输入可用 |

### 2.3 进度条 (4 步)

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 语义匹配  │ → │ SQL 生成  │ → │ 安全校验  │ → │   执行    │
└──────────┘    └──────────┘    └──────────┘    └──────────┘

状态：
  idle    — 灰点，步骤名灰色
  active  — 黄色脉动点，步骤名黄色
  done    — 绿点，步骤名绿色
  error   — 红点，步骤名红色
```

当前 `handleSSE` 中仅对 `status.message === 'starting'` 做了状态推进。正式前端应：
- 后端在 pipeline 各阶段发出带 `phase` 字段的 status 事件
- 前端按 `phase` 值（`semantic_matching` / `sql_generation` / `security_validation` / `execution`）驱动进度条

### 2.4 消息气泡类型

#### 2.4.1 用户气泡
```
┌─────────────────────────────────┐
│                        👤 你    │
│           昨天的订单总数         │
└─────────────────────────────────┘
```
- 右对齐，黄色背景 `var(--accent)`
- 头像"你"，黄色圆形

#### 2.4.2 系统气泡 — 文本
```
┌─────────────────────────────────┐
│ Q                               │
│ 正在分析你的查询…                │
└─────────────────────────────────┘
```
- 左对齐，深色卡片背景 `var(--bg-card)`
- 头像"Q"，深色圆形带边框
- 支持流式更新（同一气泡内 HTML 替换）

#### 2.4.3 系统气泡 — SQL 块
```
┌─────────────────────────────────┐
│ Q                               │
│ ┌─ SQL ───────────────────────┐│
│ │ SELECT COUNT(*)              ││
│ │ FROM orders                  ││
│ │ WHERE created_at >= '...'    ││
│ └──────────────────────────────┘│
└─────────────────────────────────┘
```
- SQL 块可折叠（右上角有展开/收起箭头）
- 深绿色等宽字体 `var(--sql-text)`，深色代码背景 `var(--sql-bg)`
- 右上角显示"SQL"标签 + 复制按钮
- 如果 SQL 来自安全校验回退，在 SQL 头部显示黄色警告条："此 SQL 经过自动修正"

#### 2.4.4 系统气泡 — 确认卡片
```
┌─────────────────────────────────┐
│ Q                               │
│ ┌─ ⚠ 需要确认以下内容 ─────────┐│
│ │                               ││
│ │ · 时间范围：本周 →            ││
│ │   2026-06-15 ~ 2026-06-21     ││
│ │                               ││
│ │ · 状态值："结了" → 已完成     ││
│ │   备选：已取消 / 待发货       ││
│ │                               ││
│ │ · 模糊量化："大额" 指什么     ││
│ │   范围？请输入金额阈值         ││
│ │   [___输入框___]              ││
│ │                               ││
│ │ [✓ 确认]  [✗ 取消查询]       ││
│ └───────────────────────────────┘│
└─────────────────────────────────┘
```

**确认卡片交互规则**：

1. 每个 need_confirm item 的渲染取决于 `reason` 类型：
   - `time_normalized` — 展示原始输入 → 标准化时间范围，无交互修改
   - `enum_matched` — 展示原始值 → 匹配结果，下方列出备选值供点击切换
   - `region_resolved` — 展示原始地名 → 解析后的区域范围
   - `name_matched` — 展示简称 → 全称映射，列出备选
   - `quantifier_fuzzy` — 展示模糊词，提供输入框让用户键入具体阈值

2. 用户可选择单项修改（点击备选值 / 修改输入框内容）
3. 「确认」按钮 → POST `/api/query/{id}/confirm` { confirmed: true }
4. 「取消」按钮 → POST `/api/query/{id}/cancel`，清除当前查询

**注意**：当前后端 confirm/cancel 端点返回 `"status": "acknowledged"`，未实际恢复/中止 pipeline。正式前端需等待后端实现真实的审核断点续传。

#### 2.4.5 系统气泡 — 结果表格
```
┌─────────────────────────────────┐
│ Q                               │
│ ┌─ 查询结果 ─── 12 行 ─────────┐│
│ │ ┌──────────┬──────────┬─────┐ ││
│ │ │  日期     │ 订单数    │ 金额 │ ││
│ │ ├──────────┼──────────┼─────┤ ││
│ │ │ 2026-06  │ 1,234    │ 56万 │ ││
│ │ │ 2026-05  │ 1,100    │ 48万 │ ││
│ │ └──────────┴──────────┴─────┘ ││
│ │                               ││
│ │ 📊 本月共 1,234 笔订单，      ││
│ │    金额合计 56 万元            ││
│ └───────────────────────────────┘│
└─────────────────────────────────┘
```
- 表格上方：列数 × 行数 统计
- 表格：黄色表头行，深色数据行，支持水平滚动（列多时）
- 表格下方：LLM 生成的查询结果摘要（仅聚合查询时显示）
- 右下角：复制 CSV 按钮

#### 2.4.6 系统气泡 — 错误
```
┌─────────────────────────────────┐
│ Q                               │
│ ⚠ 查询执行失败                  │
│                                 │
│ 错误详情：Table 'orders_2025'   │
│ 不存在。系统已自动触发元数据     │
│ 同步，请稍后重试。              │
│                                 │
│ [🔄 重试]                       │
└─────────────────────────────────┘
```
- 红色边框和图标
- 如果后端触发自愈（metadata sync / auto-join），显示自愈状态
- 「重试」按钮重新发起同一查询

### 2.5 输入区交互

```
┌──────────────────────────────────────────────────┐
│  输入查询，例如「昨天的订单总数」…    [📎] [↑]  │
└──────────────────────────────────────────────────┘
```

- **Enter** → 发送（Shift+Enter 换行）
- **发送按钮** → 黄色箭头图标，运行时变为灰色禁用态 + 旋转动画
- **空状态快捷提示**：4 个可点击的示例查询 pill，点击后自动填入输入框并发送
- **历史回填**：在输入框中按 ↑ 键可回溯上一条消息（仅在焦点在空输入框时）
- **粘贴支持**：支持粘贴多行文本

### 2.6 多步查询展示（V2 规划）

当前项目多步执行（`agents/graph.py`）为 dry-run 占位。正式实现后：

```
┌─────────────────────────────────┐
│ Q                               │
│ 检测到复杂查询，已拆解为 3 步：  │
│                                 │
│ Step 1 ✅ 本月订单总数          │
│   → 12,345 笔                   │
│ Step 2 ✅ 上月订单总数          │
│   → 10,200 笔                   │
│ Step 3 ⏳ 计算环比变化          │
│   → 增长 21.0%                  │
│                                 │
│ 📊 汇总：本月环比增长 21.0%     │
└─────────────────────────────────┘
```

每个子步骤有独立的状态图标（⏳运行中 / ✅完成 / ❌失败），失败步骤可单独重试。

## 三、会话管理

### 3.1 会话生命周期

```
新建会话 → 多轮对话 → 自动过期 (30min TTL)
                │
                ├── 手动切换会话
                └── 会话列表显示标题（首条查询截断）
```

### 3.2 会话列表

```
┌────────────────────────────┐
│  昨天的订单总数             │  ← 截断首条查询
│  14:32                     │
├────────────────────────────┤
│  本月销售额 Top 10          │
│  14:28                     │
├────────────────────────────┤
│  华为的合同                 │  ← 当前激活 (黄色高亮)
│  13:50                     │
└────────────────────────────┘
```

- 每个会话项显示：标题 = 首条查询前 24 字 + 创建时间
- 当前激活的会话黄色高亮
- 点击切换 → 重新加载该会话的历史消息
- **当前限制**：切换会话时 messages 区域仅显示 empty state，历史消息未从 Redis 回填。正式前端应调用 `GET /api/conversations/{session_id}` 获取 turns 并渲染。

### 3.3 会话自动创建

- 发送第一条查询时，如果没有 sessionId，自动 `POST /api/session`
- 后端 SSE 流响应头 `X-Session-Id` 返回新会话 ID
- 或在页面加载时预创建会话（当前行为）

## 四、管理控制台

### 4.1 布局

管理控制台在当前原型中以 CSS `display:none/block` 切换，与对话页互斥显示。

**建议改进**：改为侧边栏「管理」Tab 打开后以全宽覆盖方式展示，顶部有返回按钮。6 个卡片垂直排列，每个卡片可折叠。

### 4.2 卡片交互规范

```
┌─ 知识图谱 ─────── 12表 / 48列 / 15边 ─┐
```

交互：
- 当前用 tag 展示节点/边——标签式展示，适合数据量小时
- **V2 建议**：改为 D3.js / vis-network 力导向图，可拖拽节点、双击展开字段、点击边显示详情

```
│                                            │
```

## 九、与后端协作的接口约定

### 9.1 当前后端 MVP 限制

以下功能当前后端为 stub / 占位，前端应做好降级处理：

| 功能 | 后端现状 | 前端降级方案 |
|------|---------|-------------|
| confirm/cancel 查询 | 返回 acknowledged，不实际断点续传 | 确认后仅更新气泡文字，不期望后续 SSE |
| 多步查询执行 | dry-run 占位 | 不展示多步进度，当作单步处理 |
| 错误自愈 | healing 模块未接入 pipeline | 只显示错误信息，不展示自愈进度 |
| 会话历史回填 | Redis 有数据但前端未调用 | 切换会话时调用 GET conversations/{sid} 并渲染 |
| 数据源管理页 | 无前端页面 | 本文档设计了完整页面，需新建 |
| 知识图谱可视化 | 仅有数据接口 | 先用 tag 展示，预留 D3.js 接口 |
| 审核策略 | 内存存储 | 直接使用当前 PUT 接口 |

### 9.2 后端需要暴露但当前缺失的信息

以下信息前端需要但后端 SSE 当前未区分发送，建议后端增加：

1. **阶段状态事件** — 当前仅有 `status: "starting"`，建议增加 phase 字段：
   ```
   status: { phase: "semantic_matching" | "sql_generation" | "security_validation" | "execution" }
   ```
2. **SQL 文本** — 当前仅在 result 中附带，建议单独发送 `sql` 事件：
   ```
   sql: { text: "SELECT ...", warnings: [] }
   ```
3. **need_confirm item 类型** — 建议在 item 中增加 `type` 字段（time/enum/region/name/quantifier），前端据此选择不同的交互控件
4. **当前激活的数据源 ID** — 管理端 API 需要 data_source_id，建议提供 `GET /api/admin/active-data-source` 端点

## 十一、技术选型建议

当前原型为单文件 HTML (566 行)，以下为正式前端的建议：

### 方案 A：渐进增强（推荐快速迭代）

- 保持单文件架构，拆分 CSS/JS 到独立文件
- 引入 **Alpine.js** 或 **Petite-Vue** 做响应式状态管理
- 引入 **HTMX** 处理管理端 CRUD（避免手写 fetch 样板）
- 优点：零构建、可直接在浏览器打开、与后端 FastAPI 配合自然

### 方案 B：Next.js 重写（推荐长期维护）

- 按设计文档指定的 Next.js 方案重写
- **React 18 + TypeScript + Tailwind CSS**
- SSE 使用 `EventSource` API 或 `fetch` + ReadableStream
- 管理端使用 **shadcn/ui** 组件库
- 图谱可视化：**D3.js** 或 **vis-network**
- 优点：类型安全、组件复用、生态丰富

### 推荐

V1 阶段先用方案 A 快速补齐缺失页面（数据源管理），方案 B 在 V2 阶段实施。
